# 超大 GeoTIFF 怎么跑 SAM？瓦片化整图处理的工程实践

## 导语

普通图片可以直接送进 SAM，但遥感影像不行。

一张城市级 GeoTIFF 动辄几万像素宽高。如果直接整图推理，不仅显存扛不住，浏览器和后端内存也会被拖垮。

因此，超大影像处理的核心不是“把模型换大”，而是设计一条稳定的瓦片化处理链路。

本文介绍 GeoAI SAM 平台中的整幅影像处理方案。

## 为什么不能整图送入 SAM

SAM 对输入分辨率很敏感。影像越大，显存、推理时间和中间张量都会快速增加。

遥感场景里常见问题包括：

- 影像尺寸过大，无法一次性加载到 GPU。
- 目标尺度不一致，整图缩放会损失小目标。
- 推理结果很难直接回写地理坐标。
- 单次失败会导致整幅影像任务重跑。

所以工程上更可控的方案是：

```text
大 GeoTIFF
  -> 切成重叠瓦片
  -> 每个瓦片独立推理
  -> 每个瓦片独立矢量化
  -> 合并所有矢量结果
```

## 瓦片化处理流程

平台中的整图任务是一个后台线程：

```text
POST /api/process/full
  -> 创建 task_id
  -> 后台遍历瓦片
  -> 前端轮询 /api/process/status
```

核心步骤如下：

```text
1. 打开 GeoTIFF
2. 根据 tile_size 和 overlap 生成窗口
3. 每个窗口写出临时瓦片 GeoTIFF
4. 对瓦片执行 text / auto / point / box 推理
5. Mask 后处理
6. Mask 矢量化
7. 合并 GeoDataFrame
8. 输出 GeoPackage / GeoJSON / Shapefile
```

这样即使原图很大，每次进入模型的也只是一个固定大小的瓦片。

## tile_size 和 overlap 怎么选

两个参数最关键：

| 参数 | 推荐值 | 作用 |
| --- | --- | --- |
| `tile_size` | 2048 | 单个瓦片边长 |
| `overlap` | 256 | 相邻瓦片重叠区域 |

`tile_size` 越大，上下文越完整，但显存压力越高。  
`overlap` 越大，跨瓦片目标越不容易被切断，但重复计算更多。

对于 6GB 显存，推荐从：

```text
tile_size = 2048
overlap = 256
```

开始。如果显存不足，把 `tile_size` 降到 1024。如果边界断裂明显，把 `overlap` 提到 384 或 512。

## 不同模式的整图行为

整图处理不是简单把交互标注重复 255 次。不同模式的逻辑不同。

| 模式 | 处理方式 | 适合场景 |
| --- | --- | --- |
| 文本 | 遍历所有瓦片 | 建筑、水体、道路等同类目标提取 |
| 自动 | 遍历所有瓦片 | 发现候选目标 |
| 点提示 | 只处理包含提示点的瓦片 | 局部区域精修 |
| 框提示 | 只处理与框相交的瓦片 | 框定范围内提取 |

因此，如果目标是“处理整幅影像里的所有建筑物”，优先使用文本模式，而不是点模式。

## 如何保留地理坐标

每个瓦片写出时，都保留自己的 transform：

```python
profile.update(
    width=window.width,
    height=window.height,
    transform=rasterio.windows.transform(window, src.transform),
)
```

这样瓦片上的 Mask 被矢量化后，生成的 Polygon 仍然在原始影像坐标系中。

后续合并时，不需要手动平移几何，只要保证每个瓦片 GeoTIFF 的地理参考正确即可。

## 为什么要做进度任务

整图处理可能持续数分钟到数小时，不能用普通同步 HTTP 请求。

平台采用任务状态结构：

```json
{
  "task_id": "...",
  "status": "running",
  "progress": 0.37,
  "done_tiles": 94,
  "total_tiles": 255,
  "message": "处理瓦片 95/255"
}
```

前端每隔几秒轮询：

```text
GET /api/process/status?task_id=...
```

这样用户能看到当前阶段、百分比和瓦片计数。

文本模式首次运行时，模型加载可能比较慢。此时进度会显示：

```text
正在加载文本标注模型
```

避免用户误以为系统没有处理。

## 输出格式怎么选

整图结果可能包含大量 Polygon。推荐使用 GeoPackage：

| 格式 | 适用场景 |
| --- | --- |
| GeoPackage | 单文件，适合大结果和 GIS 软件 |
| GeoJSON | 便于 Web 查看和调试 |
| Shapefile | 兼容老 GIS 流程，但文件多、字段受限 |

如果结果很大，不建议默认用 GeoJSON。

## 常见踩坑

### 1. 瓦片边界出现断裂

原因通常是 overlap 太小。增大重叠像素，或在后处理时做合并和面积过滤。

### 2. 显存不足

降低 `tile_size`，或者改用 `vit_b`。

### 3. 处理进度一直是 0%

可能正在加载文本模型或下载权重。需要把模型缓存固定在项目 `models/` 目录，并在任务消息中展示加载阶段。

### 4. 结果为空

先判断是模型没有识别，还是提示模式不适合。比如点模式只处理包含提示点的瓦片，不会自动遍历全图目标。

## 小结

超大 GeoTIFF 跑 SAM 的关键是工程拆分：

- 用 TiTiler 解决浏览器显示。
- 用窗口切片解决显存限制。
- 用 transform 保留地理坐标。
- 用任务队列和进度轮询解决长耗时。
- 用矢量合并生成最终 GIS 标签。

大影像不是不能处理，而是不能当普通图片处理。
